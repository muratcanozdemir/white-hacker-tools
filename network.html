dig dns
dig example.com A dig example.com ANY +noall +answer dig @8.8.8.8 example.com A dig +trace example.com dig -x 93.184.216.34 dig example.com A +short +stats dig _http._tcp.example.com SRV
+trace follows the full delegation chain from root — the definitive way to diagnose propagation issues. @8.8.8.8 bypasses local resolver to test upstream. SRV lookups are essential for Kubernetes service discovery and Consul health checks.
drill dns
drill example.com A drill -T example.com drill -D example.com drill -k /etc/trusted-key.key example.com drill @1.1.1.1 example.com AAAA
DNSSEC-aware alternative to dig from ldns. -T traces the full chain; -D validates DNSSEC signatures. Preferred over dig when debugging DNSSEC validation failures — clearer chain-of-trust output.
systemd-resolve / resolvectl dns
resolvectl status resolvectl query example.com resolvectl dns resolvectl flush-caches resolvectl statistics resolvectl monitor
Inspect and control systemd-resolved on Linux — see per-interface DNS servers, flush the cache, watch live query traffic with monitor. Essential first stop when DNS behaves differently from dig vs application — the stub resolver may be the culprit.
ss tcp / udp
ss -tlnp ss -tlnp sport = :8080 ss -tnp state established ss -s ss -tnp dst 10.0.0.1 ss -tlnp | awk 'NR>1 {print $5, $7}'
Replaces netstat — faster, more detailed socket stats. -tlnp: TCP, listening, numeric, process. -s gives aggregate socket statistics (TIME_WAIT count etc). Filters by state, src, dst without grep.
mtr tcp / udp
mtr --report example.com mtr --report --report-cycles=20 example.com mtr --tcp --port 443 example.com mtr --udp --port 53 8.8.8.8 mtr --json example.com | jq '.report.hubs[] | select(.Loss > 0)'
Combines traceroute and ping — shows per-hop latency and packet loss continuously. --tcp mode tests reachability at the application layer (not just ICMP), which many firewalls pass differently. --json output feeds automated latency regression checks.
ip / iproute2 tcp / udp
ip addr show ip route get 8.8.8.8 ip route show table all ip -s link show eth0 ip neigh show ip netns list ip netns exec <ns> ss -tlnp
route get shows the exact route a packet will take — which interface, gateway, src IP. netns exec runs any command inside a network namespace — essential for debugging container networking without entering the container.
iperf3 tcp / udp
iperf3 -s iperf3 -c <server> -t 30 iperf3 -c <server> -u -b 100M -t 30 iperf3 -c <server> -P 8 -t 30 iperf3 -c <server> -R -t 30 iperf3 -c <server> --json | jq '.end.sum_received.bits_per_second / 1e9'
Measure raw TCP/UDP throughput between two hosts. -P 8 parallel streams saturates multi-queue NICs for accurate bandwidth measurement. -R reverses direction (server → client). Run inside K8s pods to validate CNI plugin throughput between nodes.
curl (deep-dive) http
curl -v https://example.com 2>&1 | grep -E "^[<>*]" curl -w "@curl-format.txt" -o /dev/null -s https://example.com curl --http2 -I https://example.com curl --http3 https://example.com curl -x http://proxy:3128 https://example.com curl --resolve example.com:443:93.184.216.34 https://example.com curl -D - -o /dev/null -s https://example.com curl --compressed -H "Accept-Encoding: gzip" https://example.com/api
-w "@curl-format.txt" with a timing template gives DNS, TCP, TLS, TTFB breakdown per request — the right tool before reaching for a profiler. --resolve overrides DNS for a specific host:port — test a new backend IP without touching /etc/hosts. -D - dumps response headers to stdout.
curl-format.txt http
# save as curl-format.txt, then: curl -w "@curl-format.txt" -o /dev/null -s URL namelookup: %{time_namelookup}s connect: %{time_connect}s appconnect: %{time_appconnect}s pretransfer: %{time_pretransfer}s redirect: %{time_redirect}s starttransfer: %{time_starttransfer}s ---------- total: %{time_total}s size_download: %{size_download} bytes http_code: %{http_code}
Commit this file to your dotfiles or repo tooling. Breaks total request time into DNS lookup, TCP handshake, TLS handshake (appconnect - connect), server processing (starttransfer - pretransfer), and transfer. Pinpoints exactly which layer is slow.
httpie http
http GET https://api.example.com/users Authorization:"Bearer $TOKEN" http POST https://api.example.com/users name=alice role=admin http --follow --all GET https://example.com/redirect http --print=HhBb GET https://example.com http --session=./session.json POST https://api.example.com/login http --offline GET https://example.com/api
Human-readable HTTP client — syntax highlighting, JSON auto-formatting, session persistence. --print=HhBb controls what's shown: request/response headers and body. --offline prints the request without sending — useful for debugging request construction.
websocat http
websocat ws://localhost:8080/ws websocat wss://api.example.com/ws echo '{"type":"ping"}' | websocat ws://localhost:8080/ws websocat -v ws://localhost:8080/ws websocat --ping-interval=5 wss://api.example.com/ws
curl for WebSockets — connect, send messages, and inspect frames from the terminal. The gap most teams hit when they can't debug WebSocket traffic via curl alone. -v shows the HTTP upgrade handshake and raw frames.
grpcurl grpc
grpcurl -plaintext localhost:50051 list grpcurl -plaintext localhost:50051 describe mypackage.MyService grpcurl -plaintext -d '{"name":"alice"}' localhost:50051 mypackage.MyService/GetUser grpcurl -H "Authorization: Bearer $TOKEN" -d @ localhost:50051 mypackage.MyService/GetUser < payload.json grpcurl -proto myservice.proto -d '{}' localhost:50051 mypackage.MyService/List grpcurl -v -plaintext localhost:50051 grpc.health.v1.Health/Check
curl for gRPC — list services, describe methods, send requests. Works with server reflection (no proto files needed) or -proto flag. grpc.health.v1.Health/Check tests the standard health protocol that K8s probes use. -d @ reads payload from stdin.
grpc-health-probe grpc
grpc-health-probe -addr=localhost:50051 grpc-health-probe -addr=localhost:50051 -service=mypackage.MyService grpc-health-probe -addr=localhost:50051 -tls -tls-ca-cert=ca.crt # in K8s liveness probe: # exec: { command: ["/bin/grpc-health-probe", "-addr=:50051"] }
Implements the gRPC health checking protocol as a standalone binary — used as a K8s exec liveness/readiness probe for gRPC services that can't use HTTP probes. Copy the static binary into your container image at build time.
openssl s_client tls
openssl s_client -connect example.com:443 </dev/null openssl s_client -connect example.com:443 -servername example.com </dev/null 2>&1 | openssl x509 -noout -dates -subject -issuer openssl s_client -connect example.com:443 -tls1_2 </dev/null openssl s_client -connect example.com:443 -alpn h2 </dev/null | grep ALPN openssl s_client -starttls smtp -connect mail.example.com:587 echo | openssl s_client -connect example.com:443 2>&1 | grep "Cipher is"
Raw TLS handshake inspection — verify cert chain, expiry, negotiated cipher and protocol version, ALPN. -servername sends SNI (required for vhosts). -starttls smtp works for STARTTLS protocols. Pipe through x509 for clean cert parsing.
testssl.sh tls
testssl.sh example.com testssl.sh --severity HIGH example.com testssl.sh --json --jsonfile result.json example.com testssl.sh --protocols example.com testssl.sh --cipher-per-proto example.com testssl.sh --client-simulation example.com
Comprehensive TLS/SSL server assessment — cipher suites, protocol versions, HSTS, certificate transparency, OCSP stapling, known vulnerabilities (POODLE, BEAST, ROBOT). --client-simulation shows what each real browser would negotiate. Self-contained bash script; no install needed.
step CLI tls
step certificate inspect https://example.com step certificate inspect cert.pem --format json | jq '.validity' step certificate verify cert.pem --roots ca.crt step ca init step ca certificate example.com cert.pem key.pem step certificate create myservice cert.pem key.pem --profile leaf --ca ca.crt --ca-key ca.key --not-after 24h
Human-friendly PKI toolchain — inspect certs at a URL or from file, verify chains, issue short-lived certs from a local CA. Replaces ad-hoc openssl x509 pipelines. step ca init bootstraps a full internal CA in minutes — useful for mTLS in dev/test environments.
k6 load
k6 run script.js k6 run --vus 50 --duration 60s script.js k6 run --out json=results.json script.js k6 run --out influxdb=http://localhost:8086/k6 script.js k6 run --env BASE_URL=https://staging.example.com script.js # script.js stages example: # stages: [{duration:'2m',target:100},{duration:'5m',target:100},{duration:'2m',target:0}]
Scripted load testing with ramp-up/ramp-down stages, thresholds, and checks. Scripts in JS but executes in Go — no browser overhead. --out influxdb streams metrics to Grafana in real time. Define SLO thresholds in the script; k6 exits non-zero when breached.
vegeta load
echo "GET https://example.com/" | vegeta attack -rate=200 -duration=60s | vegeta report echo "GET https://example.com/" | vegeta attack -rate=500 -duration=60s | vegeta report -type=hist[0,5ms,10ms,50ms,100ms,500ms] vegeta attack -targets=targets.txt -rate=100 -duration=30s | vegeta encode | vegeta report echo "GET https://example.com/" | vegeta attack -rate=100 -duration=30s | vegeta plot > plot.html
Constant-rate HTTP load testing — unlike k6 or wrk, vegeta holds an exact request rate regardless of latency. This is the correct model for SLO validation: test whether the service meets p99 latency at the target RPS, not how fast it can go.
oha load
oha https://example.com oha -n 10000 -c 100 https://example.com oha -z 30s -c 50 https://example.com oha --http2 -n 1000 https://example.com oha -m POST -H "Content-Type: application/json" -d '{"k":"v"}' https://api.example.com/
Written in Rust — real-time TUI with live latency histogram, p50/p95/p99, and requests/sec display as the test runs. HTTP/2 support. Replaces hey and wrk for quick ad-hoc benchmarks where you want visual feedback without writing a script.
